Module system

Modules are Plain Old JavaScript Objects with some extra features:

  • automatic dependencies resolving,
  • security subsytem,
  • data model - it's a module's persistence manager. It provides access to the data obtained from a data source (relational database, NoSQL database, etc).
Modules can run on a client as well as on a server.
The code snippet below shows a simple module example:

                        
    function Documents {
        var model = this.model;
        this.logDocuments = function(categoryId) {
            model.params.catId = categoryId; // The model's data is requeried on parameter's change
            model.documents.forEach(function(doc) {
                Logger.info("Document name: " + doc.Name);
            });
        }
    }

Security

Security support comes out-of-the-box in Platypus.js. Every resource like module or SQL query can be protected on the base of a currently logged-in user's roles. Platypus.js security subsystem is based on the Java EE security model and provides a very configurable mechanism for restricting access.
Various user information storages are supported. Use database user information storage or any external provider like Microsoft Active Directory or an OpenLDAP server.

Security annotations in JS modules

                            /**Any Platypus module starts with the JsDoc annotation
 * @author athorName
 * @public - to allow usage from the client side
 * @rolesAllowed admin, user0 - to set user roles allowed to use this module
 */
 
function myModuleName() {
    var self = this;
    
    //You can define allowed roles before each function
    /*
     * @rolesAllowed admin
     */
    function adminIsGreat() {
        Logger.info("Admin is a great man!");
        if (self.principal.hasRole('developer'))
            Logger.info("Ah, you are God!;)");
    }
}

Multi-threading in JavaScript

Asynchronous coding style for the server-side logic can make your programs very complex, hard to read and maintain, leading to so called 'callback hell'. More traditional coding model with blocking IO allows to write much more cleaner and simpler code. We think that a developer should have a choice to choose the IO model for the concrete situation and to be allowed to use true multi-threading.
In Platypus.js server JavaScript works in multi-threading environment providing performance is good enough for the most types of applications (don't forget all UI logic runs on the client side). On the server side asynchronous IO model implementation is upcoming in future releases.

Here is an example of thread creating:

                          var fLock = new Lock; //use it as a semafor;
    
    for (var i = 0; i <= 10; i++)
    (function() {
            myBackgroundTask(i);
        }).invokeBackground();//To execute function "myBackgroundTask" in another thread
    
    function myBackgroundTask(someParams) {
        //some useful code here
        fLock.lock(); //in case when semafor isn't locked thread continue to execute, in the other case it'll wait for unlock
        //some more useful code here :)
        fLock.unlock(); //unlocks the a semafor
        (function() {
                doSomethingReallyAmazing();
            }).invokeLater(); //launch function "doSomethingReallyAmazing" from the background thread in UI thread, it will not stop the current thread while executing
    }

And here is an example of the asynchronous data-query execution: 

                      function doItInCallback() {
        //some code here
    }
    
    //let's query data from a database and execute function "doItInCallback" after recieving data
    model.myDataSet.requery(doItInCallback,

        }); //the first param used as the succesfull callback function and the second as the failure callback;
    
    doSomeMoreWork() //it will be executed immideatly after calling the above requery function without waiting for it's result

Universal UI forms

A form is a kind of a module with UI layout definition in a XML file.
Single-page web application or Java SE desktop application share the same forms. A library of widgets includes a bunch of simple controls, panels and menus as well as smart model widgets which work directly with the model. With visual form editor you can create your application user interface without any coding and knowing about HTML and CSS.

Data is on your fingertips

Use databases native language ï¿½ SQL. You can say exactly what you want from a database: how query data, insert or update. In our platform we go further: Platypus.js can formulate data change commands on the base of SELECT data queries. You can write database-specific queries if you want. ORM subsystem transforms relational data into JavaScript objects.

For example you have an SQL query:

/**
 * @author greatMan
 * @name testQuery
 * @public
 * @writable employees - this annotation tells Platypus what table could be modified
 */
 
 select t1.employee_id, t1.employee_name, t2.city_name
 from employees t1
 inner join cities t2 on t1.city_id = t2.city_id

This query is connected with module data model. And it's became a new JS object. It's time to do a lot of great things with it.

/**
 * @author oneGreatMan
 * @public
 */
 
function employeeUtils() {
    var self = this, model = this.model, form = this; //this line was automaticly generated on module creation
    var query = model.testQuery; //we could define any other name to this query in module data model;
    
    function processEmployees() {
        query.beforeFirst(); //each datasource have data cursor. You can control it's position by next commands: first, next, prev, last, beforeFirst and etc.
        while (query.next()) {
            if (isEmployeeBad) {
                query.deleteRow()
            } else {
                query.employee_name = getNewEmployeeName(query.employee_name);
            }
        }
        model.save(); //it will save all data changes at exactly same sequence that it was made
    }
}
